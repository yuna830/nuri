
## 사용 목적

경찰청 Safe182 API에서 수집한 실종 경보 및 실종자 검색 데이터를 안전하게 관리하기 위한 기준을 정리한다.

이 문서는 실종자 데이터를 RAG에 직접 넣기 위한 변환 규칙이 아니라, 실종자 데이터의 수집, 저장, 중복 처리, 사진 저장, 보안 관리, RAG 제외 기준을 정리하기 위한 관리 문서다.

## 기본 원칙

실종자 데이터는 개인정보성 정보가 포함될 수 있으므로 원본 데이터를 Obsidian이나 RAG에 대량 저장하지 않는다.

실종자 데이터는 서비스 화면 제공을 위해 DB에 저장하고, RAG에는 API 구조와 관리 규칙만 포함한다.

사진 Base64 원문은 DB, Obsidian, RAG에 저장하지 않는다.

인증 정보는 코드, 문서, GitHub, 캡처 이미지에 직접 작성하지 않는다.

## 전체 관리 흐름

1. 서버에서 경찰청 Safe182 API를 호출한다.
    
2. XML 응답을 수신한다.
    
3. 한글 인코딩 문제를 보정한다.
    
4. 실종자 정보 목록을 파싱한다.
    
5. 필요한 필드만 추출한다.
    
6. 사진 Base64 데이터가 있으면 이미지 파일로 저장한다.
    
7. DB에는 사진 원문이 아니라 파일 경로만 저장한다.
    
8. externalKey를 생성하여 중복 여부를 판단한다.
    
9. 기존 데이터가 있으면 갱신하고, 없으면 새로 저장한다.
    
10. syncedAt 값을 현재 시간으로 갱신한다.
    
11. RAG에는 실종자 원본 데이터가 아니라 관리 규칙만 포함한다.

## RAG 제외 이유

실종자 원본 데이터는 RAG에 직접 넣지 않는다.

이유:

1. 실종자 데이터는 개인정보성 정보가 포함될 수 있다.
    
2. 사진 데이터가 포함될 수 있다.
    
3. 최신성이 중요하다.
    
4. 오래된 데이터가 RAG 답변에 섞일 수 있다.
    
5. 원본 데이터가 대량으로 들어가면 검색 품질이 떨어질 수 있다.
    
6. 사용자 질문에 실종자 개인정보가 부적절하게 노출될 수 있다.
    

## 중복 저장 기준

경찰청 API 응답에는 우리 서비스에서 바로 사용할 내부 ID가 없을 수 있으므로 주요 필드를 조합하여 externalKey를 만든다.

```txt
중복 기준 = externalKey
```

externalKey 구성:

```txt
externalKey = 이름 + 발생일자 + 발생주소 + 성별
```

API 필드 기준:

```txt
externalKey = nm + occrde + occrAdres + sexdstnDscd
```

## 중복 처리 방식

```txt
기존 데이터 있음 → 기존 데이터 갱신
기존 데이터 없음 → 새 데이터 저장
```

저장할 때마다 syncedAt 값을 갱신한다.

```txt
syncedAt = 현재 동기화 시간
```

## 사진 데이터 관리 기준

경찰청 API의 사진 데이터는 Base64 형태로 전달될 수 있다.

사진 Base64 원문은 DB에 저장하지 않는다.

처리 방식:

1. tknphotoFile 값 확인
    
2. 값이 비어 있으면 사진 없음으로 처리
    
3. Base64 디코딩
    
4. 이미지 파일로 저장
    
5. DB에는 파일 경로만 저장
    

## 사진 저장 위치

```txt
uploads/police-missing
```

## 사진 파일명 기준

사진 파일명은 externalKey의 hash 값을 이용해 생성한다.

```txt
파일명 = externalKey hash + .jpg
```

예시:

```txt
uploads/police-missing/1a2b3c4d.jpg
```

## 사진 저장 시 주의사항

사진 Base64 원문은 Obsidian에 저장하지 않는다.

사진 Base64 원문은 RAG에 저장하지 않는다.

사진 Base64 원문은 DB에 직접 저장하지 않는다.

사진 파일은 서버 업로드 폴더에 저장하고, DB에는 경로만 저장한다.

Base64 디코딩 실패 시 사진 없이 저장할 수 있다.

## 인코딩 관리 기준

경찰청 API 응답은 한글 인코딩 문제가 발생할 수 있다.

따라서 서버에서 인코딩 보정 로직을 적용한다.

처리 후보:

```txt
UTF-8
MS949
EUC-KR
mojibake 보정 문자열
```

가장 한글이 정상적으로 보이는 문자열을 선택하여 XML 파싱에 사용한다.

## XML 파싱 기준

XML 응답에서 아래 값을 우선 확인한다.

```txt
result
msg
totalCount
list
```

정상 응답일 때만 list 데이터를 파싱한다.

## 리턴 코드 관리

|코드|메시지|처리 기준|
|---|---|---|
|00|OK|정상 처리|
|80|일일 조회 건수 초과|수집 중단 및 로그 기록|
|99|각 누락 항목|요청 파라미터 확인 후 실패 처리|

## 요청 제한 관리

경찰청 API는 일일 조회 건수 제한이 있을 수 있다.

따라서 무분별한 반복 호출을 피한다.

관리 기준:

```txt
필요한 범위만 조회
페이지 수 제한
동기화 주기 설정
실패 시 즉시 무한 재시도 금지
일일 조회 건수 초과 시 수집 중단
```

## 페이지네이션 관리

경찰청 API는 page와 rowSize를 사용하여 목록을 조회한다.

기본 처리 기준:

```txt
rowSize = 100
page = 1부터 시작
```

반복 종료 조건:

1. 응답 목록이 비어 있는 경우
    
2. 현재 페이지까지 조회한 수가 totalCount 이상인 경우
    
3. 응답 목록 수가 rowSize보다 작은 경우
    
4. 서버에서 정한 최대 페이지 수에 도달한 경우

|중복 데이터 수신|기존 데이터 갱신|
|일일 조회 건수 초과|수집 중단 및 재시도 제한|

## 데이터 보관 기준

실종자 데이터는 최신성이 중요하므로 보관 기준을 정해야 한다.

권장 기준:

```txt
최근 동기화 데이터 우선 사용
syncedAt 기준으로 오래된 데이터 식별
장기간 갱신되지 않은 데이터는 화면 표시 제외 검토
삭제보다는 비활성 상태 관리 검토
```

실제 삭제 여부는 서비스 정책과 데이터 사용 목적에 따라 결정한다.

## 데이터 삭제 기준

실종자 데이터 삭제는 신중하게 처리한다.

삭제보다 상태값 또는 표시 여부로 관리하는 방식을 우선 검토한다.

예시:

```txt
isActive = false
displayYn = N
archivedAt = 보관 처리 시간
```

현재 테이블에 해당 필드가 없다면 추후 필요 시 추가할 수 있다.

## RAG 문서 작성 기준

실종자 데이터 관련 RAG 문서는 원본 데이터가 아니라 설명 문서로만 작성한다.

포함 가능한 문서:

```txt
경찰청_실종경보_API
경찰청_실종자검색_API
경찰청_API_서버_처리흐름
실종자데이터_관리규칙
```

포함하지 않는 문서:

```txt
실제 실종자 목록 Markdown
실종자 사진 Markdown
실종자 상세 원문 Markdown
```

## Qdrant 저장 기준

Qdrant에는 실종자 원본 데이터 chunk를 저장하지 않는다.

저장 가능 대상:

```txt
API 설명 문서
서버 처리 흐름 문서
DB 저장 기준 문서
보안 관리 규칙 문서
```

저장 제외 대상:

```txt
실종자 원본 목록
실종자 상세 정보
사진 Base64
개인정보성 원본 응답
```

## 최종 정리

실종자데이터 관리의 핵심은 원본 데이터와 관리 문서를 분리하는 것이다.

실종자 원본 데이터는 서비스 화면 제공을 위해 `police_missing_alerts` 테이블에 저장한다.

중복 저장은 `externalKey` 기준으로 방지한다.

사진은 Base64 원문이 아니라 이미지 파일로 저장하고, DB에는 파일 경로만 저장한다.

Obsidian과 RAG에는 실종자 원본 데이터가 아니라 API 구조, 서버 처리 흐름, DB 저장 기준, 보안 관리 기준만 포함한다.
