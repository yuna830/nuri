
## 사용 목적

공공데이터포털의 지자체복지서비스 API에서 수집한 복지서비스 데이터를 RAG 검색에 적합한 Markdown 문서로 변환하기 위한 기준을 정리한다.

이 문서는 원본 XML 데이터를 그대로 저장하거나 임베딩하지 않고, 사용자가 이해할 수 있는 설명형 문서로 변환하기 위한 규칙이다.

## 대상 API

```txt
한국사회보장정보원_지자체복지서비스
```

## 관련 API 문서

```txt
01_복지_API/한국사회보장정보원_지자체복지서비스.md
```

## 관련 서버 처리 문서

```txt
03_프로젝트_구현메모/복지_API_서버_처리흐름.md
03_프로젝트_구현메모/DB_저장_중복처리_기준.md
```

## 변환 대상 데이터

지자체복지서비스 API의 목록 조회와 상세 조회 결과를 합쳐 Markdown 문서로 변환한다.

목록 조회만으로는 지원 대상, 지원 내용, 신청 방법이 부족할 수 있으므로 가능하면 상세 조회 결과를 함께 사용한다.

## 변환 대상 Endpoint

```txt
https://apis.data.go.kr/B554287/LocalGovernmentWelfareInformations/LcgvWelfarelist

https://apis.data.go.kr/B554287/LocalGovernmentWelfareInformations/LcgvWelfaredetailed
```

## 변환 기본 원칙

원본 XML 전체를 그대로 Markdown에 넣지 않는다.

사용자 질문에 답변할 수 있는 정보만 선별하여 설명형 문서로 변환한다.

비어 있는 필드는 Markdown에 포함하지 않는다.

같은 의미의 중복 필드는 하나로 정리한다.

인증키, 내부 DB ID, 시스템 필드는 Markdown에 포함하지 않는다.

## 변환 흐름

1. 지자체복지서비스 목록 조회 API를 호출한다.
    
2. 목록 응답에서 servId를 추출한다.
    
3. servId를 이용해 상세 조회 API를 호출한다.
    
4. 목록 정보와 상세 정보를 하나의 데이터로 병합한다.
    
5. 필요한 필드만 추출한다.
    
6. 사용자 질문에 답변하기 쉬운 Markdown 구조로 변환한다.
    
7. 변환된 Markdown 문자열을 기준으로 raw_hash를 계산한다.
    
8. 기존 raw_hash와 비교한다.
    
9. 변경된 문서만 rag_documents에 갱신한다.
    
10. 변경된 문서만 Qdrant에 재임베딩한다.
    

## 문서 단위

복지서비스 1개를 Markdown 문서 1개로 변환한다.

```txt
복지서비스 1건 = Markdown 문서 1개
```

예시:

```txt
servId = LIF00001138
↓
LIF00001138.md
```

## 문서 식별 기준

복지서비스 문서는 `servId`를 기준으로 식별한다.

```txt
source_type = welfare_service
source_id = servId
```

servId가 없는 경우에는 서비스명, 소관 기관명, 담당 기관명, 상세 링크를 조합하여 대체 식별자로 사용한다.

```txt
대체 식별 기준 = servNm + jurMnofNm + jurOrgNm + servDtlLink
```

## Markdown 파일명 기준

파일명은 사람이 알아보기 쉬우면서도 중복되지 않게 만든다.

권장 형식:

```txt
복지서비스명_servId.md
```

예시:

```txt
노인맞춤돌봄서비스_LIF00001138.md
```

파일명에 사용할 수 없는 문자는 제거하거나 `_`로 변경한다.

## RAG 문서 제목 기준

Markdown 문서의 첫 번째 제목은 복지서비스명을 사용한다.

```md
# 복지서비스명
```

서비스명이 없는 경우에는 `복지서비스 정보`로 대체한다.

아래 정보는 Markdown에 포함하지 않는다.


## 상세 조회 필드 처리 기준

상세 조회 API에서 아래 정보가 제공되는 경우 Markdown에 포함한다.

```txt
지원 대상
선정 기준
지원 내용
신청 방법
구비 서류
접수 기관
문의처
온라인 신청 여부
근거 법령
```

상세 조회 필드명은 API 응답 구조에 따라 다를 수 있으므로, 실제 응답을 확인한 뒤 필드명을 매핑한다.

## Markdown 기본 구조

복지서비스 Markdown 문서는 아래 구조를 따른다.

```md
# 복지서비스명

## 서비스 요약

## 지원 대상

## 선정 기준

## 지원 내용

## 신청 방법

## 문의처

## 담당 기관

## 지원 정보

## 지역 및 분류 정보

## 상세 링크

## 우리 서비스 활용
```

## Markdown 변환 예시

```md
# 노인맞춤돌봄서비스

## 서비스 요약
일상생활 영위가 어려운 취약 노인에게 안전 확인, 생활교육, 사회참여, 일상생활 지원 등을 제공하는 복지서비스다.

## 지원 대상
돌봄이 필요한 노인 또는 취약계층 노인을 대상으로 한다.

## 선정 기준
연령, 소득, 건강 상태, 돌봄 필요도 등을 기준으로 대상 여부를 판단할 수 있다.

## 지원 내용
안전 확인, 생활교육, 사회참여 지원, 일상생활 지원 등의 서비스를 제공한다.

## 신청 방법
주민센터, 지자체 담당 부서 또는 관련 기관을 통해 신청할 수 있다.

## 문의처
문의처 정보가 제공되는 경우 작성한다.

## 담당 기관
소관 기관: 보건복지부  
담당 기관: 지자체 또는 수행기관

## 지원 정보
지원 주기: 월별 또는 상시  
제공 유형: 서비스

## 지역 및 분류 정보
생애주기: 노년  
가구 유형: 취약계층  
관심 주제: 돌봄, 생활지원

## 상세 링크
상세 링크가 제공되는 경우 작성한다.

## 우리 서비스 활용
사용자의 나이, 거주 지역, 건강 상태, 가구 유형을 기준으로 복지제도 추천과 RAG 기반 Q&A에 활용한다.
```

## 빈 값 처리 기준

API 응답 필드는 비어 있을 수 있으므로 아래 기준을 적용한다.

|상황|처리|
|---|---|
|값이 null|Markdown에서 제외|
|값이 빈 문자열|Markdown에서 제외|
|공백만 있음|trim 후 제외|
|숫자 필드 오류|저장하지 않거나 원문 문자열로 유지|
|날짜 형식 오류|원문 문자열로 유지|
|여러 값이 배열 형태|쉼표 또는 줄바꿈으로 정리|

## 중복 문장 처리 기준

동일한 내용이 여러 필드에 반복될 경우 한 번만 작성한다.

예시:

```txt
서비스 요약과 지원 내용이 거의 같음
↓
서비스 요약에는 짧게 작성
지원 내용에는 구체적으로 작성
```

## 목록 조회와 상세 조회 병합 기준

목록 조회와 상세 조회에 같은 필드가 있을 경우 상세 조회 값을 우선 사용한다.

```txt
상세 조회 값 있음 → 상세 조회 값 사용
상세 조회 값 없음 → 목록 조회 값 사용
둘 다 없음 → Markdown에서 제외
```

## 문장 변환 기준

API 응답이 짧은 단어나 코드 형태라면 사용자가 이해할 수 있는 문장으로 바꾼다.

예시:

```txt
제공 유형: 현금
↓
이 서비스는 현금성 지원 형태로 제공될 수 있다.
```

```txt
생애주기: 노년
↓
주로 노년층을 대상으로 하는 복지서비스다.
```

단, 원문 의미를 바꾸거나 없는 내용을 추측해서 추가하지 않는다.

## RAG 검색 품질을 위한 작성 기준

사용자가 실제로 질문할 만한 표현을 포함한다.

예시 질문:

```txt
내가 받을 수 있는 복지제도 알려줘
노인 복지서비스 추천해줘
송파구에서 받을 수 있는 복지 혜택 있어?
독거노인 지원 제도 있어?
장애인 대상 복지서비스 알려줘
복지서비스 신청 방법 알려줘
```

따라서 Markdown에는 아래 표현이 자연스럽게 포함되도록 작성한다.

```txt
지원 대상
신청 방법
지원 내용
지역
문의처
담당 기관
받을 수 있는 사람
이용 가능한 조건
```

## raw_hash 계산 기준

raw_hash는 최종 Markdown 문자열을 기준으로 계산한다.

```txt
raw_hash = hash(최종 Markdown 문서 전체)
```

API 원본 XML이 아니라, 실제 RAG에 들어갈 Markdown 문서를 기준으로 한다.

## raw_hash 비교 기준

기존 문서와 새 문서의 raw_hash가 같으면 재임베딩하지 않는다.

```txt
기존 raw_hash == 신규 raw_hash
↓
rag_documents 내용 변경 없음
Qdrant 재임베딩 안 함
```

기존 문서와 새 문서의 raw_hash가 다르면 문서를 갱신하고 재임베딩한다.

```txt
기존 raw_hash != 신규 raw_hash
↓
rag_documents 갱신
Qdrant 재임베딩
```

## rag_documents 저장 기준

복지서비스 Markdown 문서는 `rag_documents` 테이블에 저장한다.

저장 예시:

|필드|값|
|---|---|
|source_type|welfare_service|
|source_id|servId|
|title|복지서비스명|
|content|변환된 Markdown 전체|
|raw_hash|Markdown 기준 hash|
|source_url|상세 링크|
|created_at|최초 저장 시간|
|updated_at|마지막 갱신 시간|

## Qdrant 저장 기준

Qdrant에는 Markdown 문서를 적절한 크기로 나누어 저장한다.

문서가 너무 짧으면 하나의 chunk로 저장한다.

문서가 길면 항목 단위로 나눈다.

권장 chunk 기준:

```txt
서비스 요약 + 지원 대상
지원 내용
신청 방법 + 문의처
기관 정보 + 지역 정보
```

## chunk metadata 기준

Qdrant에 저장할 때 metadata에는 검색 필터에 필요한 정보를 포함한다.

```txt
source_type
source_id
title
region
life_stage
household_type
topic
updated_at
```

예시:

```txt
source_type = welfare_service
source_id = LIF00001138
title = 노인맞춤돌봄서비스
region = 서울특별시 송파구
life_stage = 노년
household_type = 1인가구
topic = 돌봄
```

## 지역 정보 처리 기준

지역 정보가 있는 경우 반드시 Markdown과 metadata에 포함한다.

```txt
ctpvNm = 시도명
sggNm = 시군구명
```

예시:

```md
## 지역 및 분류 정보
지역: 서울특별시 송파구
```

지역 정보가 없으면 전국 또는 지역 제한 없음으로 단정하지 않는다.

알 수 없는 경우에는 작성하지 않는다.

## 상세 링크 처리 기준

상세 링크가 있는 경우 Markdown에 포함한다.

```md
## 상세 링크
https://...
```

상세 링크가 없으면 해당 항목을 제외한다.

## 보안 관리 기준

Markdown 문서에는 인증키를 절대 포함하지 않는다.

제외 대상:

```txt
serviceKey
DATA_GO_KR_SERVICE_KEY
인증키 원문
요청 URL에 포함된 실제 serviceKey 값
```

샘플 URL을 작성해야 할 경우 아래처럼 변수 형태로 작성한다.

```txt
serviceKey={SERVICE_KEY}
```

## 변환하지 않는 경우

아래 조건에 해당하면 RAG 문서로 변환하지 않는다.

```txt
서비스명이 없는 경우
지원 내용이 전혀 없는 경우
중복 문서로 판단되는 경우
필수 식별자가 없고 대체 중복 기준도 만들 수 없는 경우
오류 응답인 경우
인증 실패 응답인 경우
```

## 오류 응답 처리 기준

API 응답이 정상인지 확인한 뒤 변환한다.

정상 응답이 아닌 경우 Markdown 문서를 만들지 않는다.

처리 예시:

```txt
resultCode가 정상 아님
응답 목록 없음
필수 필드 누락
XML 파싱 실패
```

이 경우 수집 작업 로그에 실패 사유를 기록한다.

## 수집 작업 기록 기준

복지서비스 Markdown 변환 작업은 `rag_ingest_jobs`에 기록할 수 있다.

기록 대상:

```txt
source_type = welfare_service
source_name = 한국사회보장정보원_지자체복지서비스
status = running / success / failed
total_count
success_count
failed_count
error_message
started_at

```

## 최종 정리

복지서비스 Markdown 변환은 지자체복지서비스 API의 목록 조회와 상세 조회 데이터를 사용자 질문에 답변하기 쉬운 설명형 문서로 바꾸는 과정이다.

복지서비스 1개를 Markdown 문서 1개로 변환하며, servId를 기준으로 문서를 식별한다.

RAG에는 원본 XML 전체가 아니라 서비스명, 지원 대상, 지원 내용, 신청 방법, 문의처, 기관 정보, 지역 정보 중심의 Markdown 문서만 저장한다.

raw_hash를 기준으로 변경된 문서만 갱신하고, 변경된 문서만 Qdrant에 재임베딩한다.
