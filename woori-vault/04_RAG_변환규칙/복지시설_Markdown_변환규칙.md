
## 사용 목적

공공데이터포털의 사회복지시설정보서비스 API에서 수집한 복지시설 데이터를 RAG 검색에 적합한 Markdown 문서로 변환하기 위한 기준을 정리한다.

이 문서는 원본 XML 데이터를 그대로 저장하거나 임베딩하지 않고, 사용자가 이해할 수 있는 시설 안내형 문서로 변환하기 위한 규칙이다.

## 대상 API

```txt
한국사회보장정보원_사회복지시설정보서비스
```

## 관련 API 문서

```txt
01_복지_API/한국사회보장정보원_사회복지시설정보서비스.md
```

## 관련 서버 처리 문서

```txt
03_프로젝트_구현메모/복지_API_서버_처리흐름.md
03_프로젝트_구현메모/DB_저장_중복처리_기준.md
```

## 변환 대상 데이터

사회복지시설정보서비스 API의 시설 목록 조회, 시설별 기본정보 조회, 행사 정보 조회, 구인 정보 조회 결과를 필요한 범위에서 Markdown 문서로 변환한다.

시설 데이터는 양이 많을 수 있으므로 전체 데이터를 무조건 RAG에 넣지 않는다.

사용자 지역, 시설 종류, 주요 서비스, 주소, 연락처처럼 실제 질문에 답변할 가능성이 높은 정보만 선별하여 변환한다.

## 변환 대상 Endpoint

```txt
https://apis.data.go.kr/B554287/sclWlfrFcltInfoInqirService1/getFcltListInfoInqire

https://apis.data.go.kr/B554287/sclWlfrFcltInfoInqirService1/getFcltByBassInfoInqire

https://apis.data.go.kr/B554287/sclWlfrFcltInfoInqirService1/getFcltByEvntInfo

https://apis.data.go.kr/B554287/sclWlfrFcltInfoInqirService1/getFcltByJobInfo
```

시설 종류 코드 조회 API는 시설 분류나 필터링 기준을 정리할 때 사용한다.

```txt
https://apis.data.go.kr/B554287/sclWlfrFcltInfoInqirService1/getFcltKindCodeInfoInqire
```

## 변환 기본 원칙

원본 XML 전체를 그대로 Markdown에 넣지 않는다.

시설 1개를 기준으로 사용자가 이해하기 쉬운 안내형 문서로 변환한다.

비어 있는 필드는 Markdown에 포함하지 않는다.

같은 의미의 중복 필드는 하나로 정리한다.

인증키, 내부 DB ID, 시스템 필드는 Markdown에 포함하지 않는다.

행사 정보와 구인 정보는 최신성이 중요하므로 오래된 데이터는 제외하거나 만료 처리한다.

## 변환 흐름

1. 사회복지시설 목록 조회 API를 호출한다.
    
2. 목록 응답에서 시설 코드 `fcltCd`를 추출한다.
    
3. `fcltCd`를 이용해 시설별 기본정보 조회 API를 호출한다.
    
4. 필요한 경우 행사 정보와 구인 정보 API를 추가로 호출한다.
    
5. 시설 목록 정보와 상세 정보를 하나의 데이터로 병합한다.
    
6. 필요한 필드만 추출한다.
    
7. 사용자 질문에 답변하기 쉬운 Markdown 구조로 변환한다.
    
8. 변환된 Markdown 문자열을 기준으로 raw_hash를 계산한다.
    
9. 기존 raw_hash와 비교한다.
    
10. 변경된 문서만 `rag_documents`에 갱신한다.
    
11. 변경된 문서만 Qdrant에 재임베딩한다.
    

## 문서 단위

복지시설 1개를 Markdown 문서 1개로 변환한다.

```txt
복지시설 1건 = Markdown 문서 1개
```

예시:

```txt
fcltCd = A1234567
↓
시설명_A1234567.md
```

행사 정보와 구인 정보는 양이 많거나 갱신 주기가 다를 수 있으므로 필요 시 별도 문서로 분리할 수 있다.

```txt
복지시설 기본정보 = 시설별 Markdown 문서
시설 행사 정보 = 행사별 Markdown 문서 또는 시설 문서 내 최신 행사 섹션
시설 구인 정보 = 구인별 Markdown 문서 또는 시설 문서 내 최신 구인 섹션
```

## 문서 식별 기준

복지시설 문서는 `fcltCd`를 기준으로 식별한다.

```txt
source_type = welfare_facility
source_id = fcltCd
```

`fcltCd`가 없는 경우에는 시설명, 주소, 시설 종류를 조합하여 대체 식별자로 사용한다.

```txt
대체 식별 기준 = fcltNm + addr + fcltKindNm
```

## Markdown 파일명 기준

파일명은 사람이 알아보기 쉬우면서도 중복되지 않게 만든다.

권장 형식:

```txt
시설명_fcltCd.md
```

예시:

```txt
송파노인종합복지관_A1234567.md
```

파일명에 사용할 수 없는 문자는 제거하거나 `_`로 변경한다.

## RAG 문서 제목 기준

Markdown 문서의 첫 번째 제목은 시설명을 사용한다.

```md
# 시설명
```

시설명이 없는 경우에는 `사회복지시설 정보`로 대체한다.

## 시설별 기본정보 처리 기준

시설별 기본정보 조회 API에서 아래 정보가 제공되는 경우 Markdown에 포함한다.

```txt
시설명
시설 종류
주소
전화번호
홈페이지
운영 상태
이용 대상
주요 서비스
지역 정보
위도
경도
```

시설 목록 조회와 시설별 기본정보 조회에 같은 필드가 있을 경우 상세 조회 값을 우선 사용한다.

```txt
상세 조회 값 있음 → 상세 조회 값 사용
상세 조회 값 없음 → 목록 조회 값 사용
둘 다 없음 → Markdown에서 제외
```

## 행사 정보 처리 기준

시설별 행사 정보는 최신성이 중요하므로 모든 행사 데이터를 무조건 RAG에 넣지 않는다.

포함 기준:

```txt
현재 진행 중인 행사
최근 등록된 행사
사용자에게 안내 가치가 있는 행사
시설 이용과 직접 관련된 행사
```

제외 기준:

```txt
종료된 지 오래된 행사
날짜 정보가 불분명한 행사
시설 안내와 관련성이 낮은 행사
중복 행사
```

행사 정보는 시설 문서 안에 최신 행사 섹션으로 넣거나, 필요 시 별도 문서로 관리한다.

## 구인 정보 처리 기준

시설별 구인 정보는 최신성이 중요하므로 모집 상태나 등록일 기준으로 관리한다.

포함 기준:

```txt
현재 모집 중인 구인 정보
최근 등록된 구인 정보
복지시설 일자리 안내에 필요한 정보
```

제외 기준:

```txt
모집 종료된 구인 정보
오래된 구인 정보
필수 정보가 부족한 구인 정보
중복 구인 정보
```

구인 정보는 시설 문서 안에 최신 구인 섹션으로 넣거나, 필요 시 별도 문서로 관리한다.

## Markdown 기본 구조

복지시설 Markdown 문서는 아래 구조를 따른다.

```md
# 시설명

## 시설 개요

## 위치 정보

## 연락처

## 주요 서비스

## 이용 대상

## 최신 행사 정보

## 최신 구인 정보

## 지역 및 분류 정보

## 우리 서비스 활용
```

## Markdown 변환 예시

```md
# 송파노인종합복지관

## 시설 개요
송파노인종합복지관은 지역 어르신을 대상으로 복지서비스와 생활 지원 프로그램을 제공하는 사회복지시설이다.

## 위치 정보
주소: 서울특별시 송파구 예시로 123  
지역: 서울특별시 송파구

## 연락처
전화번호: 02-0000-0000  
홈페이지: https://example.or.kr

## 주요 서비스
어르신 복지 상담, 여가 프로그램, 건강 관리 프로그램, 생활 지원 서비스를 제공할 수 있다.

## 이용 대상
지역 내 어르신 또는 복지서비스가 필요한 주민을 대상으로 한다.

## 최신 행사 정보
현재 확인 가능한 최신 행사 정보가 있는 경우 작성한다.

## 최신 구인 정보
현재 확인 가능한 최신 구인 정보가 있는 경우 작성한다.

## 지역 및 분류 정보
시설 종류: 노인복지시설  
시설 코드: A1234567

## 우리 서비스 활용
사용자의 현재 위치, 거주 지역, 복지 필요도에 따라 주변 복지시설 안내와 지역 기반 복지 추천에 활용한다.
```

## 빈 값 처리 기준

API 응답 필드는 비어 있을 수 있으므로 아래 기준을 적용한다.

|상황|처리|
|---|---|
|값이 null|Markdown에서 제외|
|값이 빈 문자열|Markdown에서 제외|
|공백만 있음|trim 후 제외|
|전화번호 없음|연락처 항목 제외|
|홈페이지 없음|홈페이지 항목 제외|
|위도·경도 없음|지도 관련 metadata 제외|
|날짜 형식 오류|원문 문자열로 유지하거나 제외|
|여러 값이 배열 형태|쉼표 또는 줄바꿈으로 정리|

## 중복 문장 처리 기준

동일한 내용이 여러 필드에 반복될 경우 한 번만 작성한다.

예시:

```txt
주소와 위치 설명이 중복됨
↓
위치 정보 섹션에 한 번만 작성
```

```txt
시설 종류와 주요 서비스 설명이 유사함
↓
시설 종류는 분류 정보에 작성하고, 주요 서비스는 설명형 문장으로 작성
```

## 목록 조회와 상세 조회 병합 기준

목록 조회와 상세 조회에 같은 필드가 있을 경우 상세 조회 값을 우선 사용한다.

```txt
상세 조회 값 있음 → 상세 조회 값 사용
상세 조회 값 없음 → 목록 조회 값 사용
둘 다 없음 → Markdown에서 제외
```

단, 목록 조회에만 존재하는 지역 정보나 시설 종류 정보는 상세 조회에 없더라도 유지한다.

## 문장 변환 기준

API 응답이 짧은 단어나 코드 형태라면 사용자가 이해할 수 있는 문장으로 바꾼다.

예시:

```txt
시설 종류: 노인복지시설
↓
이 시설은 노인복지시설에 해당한다.
```

```txt
지역: 서울특별시 송파구
↓
서울특별시 송파구에 위치한 복지시설이다.
```

단, 원문 의미를 바꾸거나 없는 내용을 추측해서 추가하지 않는다.

## RAG 검색 품질을 위한 작성 기준

사용자가 실제로 질문할 만한 표현을 포함한다.

예시 질문:

```txt
내 주변 복지시설 알려줘
송파구 복지관 있어?
노인복지시설 어디 있어?
장애인복지시설 연락처 알려줘
가까운 사회복지시설 찾아줘
복지관 행사 있어?
복지시설 구인 정보 있어?
```

따라서 Markdown에는 아래 표현이 자연스럽게 포함되도록 작성한다.

```txt
시설명
지역
주소
전화번호
시설 종류
주요 서비스
이용 대상
행사 정보
구인 정보
주변 복지시설
```

## raw_hash 계산 기준

raw_hash는 최종 Markdown 문자열을 기준으로 계산한다.

```txt
raw_hash = hash(최종 Markdown 문서 전체)
```

API 원본 XML이 아니라, 실제 RAG에 들어갈 Markdown 문서를 기준으로 한다.

## raw_hash 비교 기준

기존 문서와 새 문서의 raw_hash가 같으면 재임베딩하지 않는다.

```txt
기존 raw_hash == 신규 raw_hash
↓
rag_documents 내용 변경 없음
Qdrant 재임베딩 안 함
```

기존 문서와 새 문서의 raw_hash가 다르면 문서를 갱신하고 재임베딩한다.

```txt
기존 raw_hash != 신규 raw_hash
↓
rag_documents 갱신
Qdrant 재임베딩
```

## rag_documents 저장 기준

복지시설 Markdown 문서는 `rag_documents` 테이블에 저장한다.

저장 예시:

|필드|값|
|---|---|
|source_type|welfare_facility|
|source_id|fcltCd|
|title|시설명|
|content|변환된 Markdown 전체|
|raw_hash|Markdown 기준 hash|
|source_url|홈페이지 또는 원본 출처|
|created_at|최초 저장 시간|
|updated_at|마지막 갱신 시간|

행사 정보나 구인 정보를 별도 문서로 저장하는 경우 source_type을 구분한다.

```txt
source_type = welfare_facility_event
source_type = welfare_facility_job
```

## Qdrant 저장 기준

Qdrant에는 Markdown 문서를 적절한 크기로 나누어 저장한다.

문서가 너무 짧으면 하나의 chunk로 저장한다.

문서가 길면 항목 단위로 나눈다.

권장 chunk 기준:

```txt
시설 개요 + 시설 종류
위치 정보 + 연락처
주요 서비스 + 이용 대상
최신 행사 정보
최신 구인 정보
```

## chunk metadata 기준

Qdrant에 저장할 때 metadata에는 검색 필터에 필요한 정보를 포함한다.

```txt
source_type
source_id
title
facility_type
facility_type_code
region
ctpv_name
sgg_name
address
latitude
longitude
updated_at
```

예시:

```txt
source_type = welfare_facility
source_id = A1234567
title = 송파노인종합복지관
facility_type = 노인복지시설
facility_type_code = 001
region = 서울특별시 송파구
ctpv_name = 서울특별시
sgg_name = 송파구
address = 서울특별시 송파구 예시로 123
```

## 지역 정보 처리 기준

지역 정보가 있는 경우 반드시 Markdown과 metadata에 포함한다.

```txt
ctpvNm = 시도명
sggNm = 시군구명
addr = 상세 주소
```

예시:

```md
## 위치 정보
주소: 서울특별시 송파구 예시로 123  
지역: 서울특별시 송파구
```

지역 정보가 없으면 임의로 추측하지 않는다.

알 수 없는 경우에는 작성하지 않는다.

## 위도·경도 처리 기준

위도와 경도가 있는 경우 metadata에 포함한다.

```txt
lat = 위도
lot = 경도
```

위도와 경도는 지도 기반 주변 시설 검색에 사용할 수 있다.

단, 값이 없거나 형식이 올바르지 않으면 저장하지 않는다.

## 전화번호 처리 기준

전화번호가 있는 경우 Markdown에 포함한다.

```md
## 연락처
전화번호: 02-0000-0000
```

전화번호가 없으면 연락처 항목을 제외하거나 `전화번호 정보 없음`으로 작성한다.

사용자에게 안내할 때는 가능한 원문 값을 유지한다.

## 홈페이지 처리 기준

홈페이지 주소가 있는 경우 Markdown에 포함한다.

```md
홈페이지: https://example.or.kr
```

홈페이지 주소가 없으면 해당 항목을 제외한다.

## 행사 정보 만료 기준

행사 정보는 오래된 데이터가 RAG 답변에 섞이지 않도록 만료 기준을 둔다.

권장 기준:

```txt
행사 종료일이 지난 경우 제외
행사 종료일이 없고 등록일이 오래된 경우 제외
최근 행사만 유지
```

정확한 기간 기준은 서비스 정책에 따라 정한다.

예시:

```txt
최근 30일 이내 행사만 RAG에 포함
```

## 구인 정보 만료 기준

구인 정보는 모집 기간이 지나면 제외한다.

권장 기준:

```txt
모집 종료일이 지난 경우 제외
모집 종료일이 없고 등록일이 오래된 경우 제외
현재 모집 중인 정보만 유지
```

예시:

```txt
최근 30일 이내 구인 정보만 RAG에 포함
```

## 보안 관리 기준

Markdown 문서에는 인증키를 절대 포함하지 않는다.

제외 대상:

```txt
serviceKey
DATA_GO_KR_SERVICE_KEY
인증키 원문
요청 URL에 포함된 실제 serviceKey 값
```

샘플 URL을 작성해야 할 경우 아래처럼 변수 형태로 작성한다.

```txt
serviceKey={SERVICE_KEY}
```

## 변환하지 않는 경우

아래 조건에 해당하면 RAG 문서로 변환하지 않는다.

```txt
시설명이 없는 경우
주소와 지역 정보가 모두 없는 경우
시설 코드가 없고 대체 중복 기준도 만들 수 없는 경우
중복 문서로 판단되는 경우
오류 응답인 경우
인증 실패 응답인 경우
사용자 질문에 답할 만한 정보가 거의 없는 경우
```

## 오류 응답 처리 기준

API 응답이 정상인지 확인한 뒤 변환한다.

정상 응답이 아닌 경우 Markdown 문서를 만들지 않는다.

처리 예시:

```txt
resultCode가 정상 아님
응답 목록 없음
필수 필드 누락
XML 파싱 실패
```

이 경우 수집 작업 로그에 실패 사유를 기록한다.

## 수집 작업 기록 기준

복지시설 Markdown 변환 작업은 `rag_ingest_jobs`에 기록할 수 있다.

기록 대상:

```txt
source_type = welfare_facility
source_name = 한국사회보장정보원_사회복지시설정보서비스
status = running / success / failed
total_count
success_count
failed_count
error_message
started_at
finished_at
```

행사 정보와 구인 정보를 별도로 수집하는 경우 source_type을 구분한다.

```txt
source_type = welfare_facility_event
source_type = welfare_facility_job
```

## 최종 정리

복지시설 Markdown 변환은 사회복지시설정보서비스 API의 시설 목록, 시설별 기본정보, 행사 정보, 구인 정보를 사용자 질문에 답변하기 쉬운 안내형 문서로 바꾸는 과정이다.

복지시설 1개를 Markdown 문서 1개로 변환하며, `fcltCd`를 기준으로 문서를 식별한다.

RAG에는 원본 XML 전체가 아니라 시설명, 시설 종류, 주소, 연락처, 주요 서비스, 지역 정보, 최신 행사 정보, 최신 구인 정보 중심의 Markdown 문서만 저장한다.

행사 정보와 구인 정보는 최신성이 중요하므로 만료 기준을 두고 관리한다.

raw_hash를 기준으로 변경된 문서만 갱신하고, 변경된 문서만 Qdrant에 재임베딩한다.
