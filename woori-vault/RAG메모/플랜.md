# Oracle VM PostgreSQL로 RAG 메타데이터 이전 및 Markdown Ingest 준비

## Summary
- `ai_backend`의 RAG 메타데이터 DB 연결을 Neon 전용 설정에서 Oracle VM PostgreSQL 설정으로 바꾼다.
- Qdrant는 그대로 벡터 저장소로 사용하고, PostgreSQL은 `rag_documents`, `rag_ingest_jobs`, `rag_public_welfare_jobs` 같은 작업/문서 메타데이터만 저장한다.
- 이후 `woori-vault/복지정책/*.md.md` 문서를 `/api/upload/markdown/ingest`로 순차 업로드해 Qdrant에 넣는다.

## Key Changes
- `ai_backend` 설정은 `DATABASE_URL` 하나로 통일한다.
  - 예시 형태: `postgresql+psycopg://woori:${OCI_DB_PASSWORD}@168.107.27.186:5432/woori`
  - 기존 `neon_database_url` 필드는 `database_url`로 교체한다.
  - `metadata_service.py`, `public_welfare_job_service.py`는 `settings.database_url`을 사용한다.
- Oracle VM PostgreSQL에 RAG 메타 테이블을 만든다.
  - `rag_documents`: 문서 ID, 파일명, 제목, 출처, 상태, Qdrant 컬렉션, text 길이, chunk 수, 생성/수정 시각
  - `rag_ingest_jobs`: 파일 ingest job 상태, chunk 처리 수, 저장 수, cache hit/miss, 에러 메시지
  - `rag_public_welfare_jobs`: 공공복지 API sync job 상태
- 테이블 DDL은 Spring의 `schema-update.sql`에 추가하는 방향으로 간다.
  - Spring이 이미 Oracle VM PostgreSQL을 보고 있고 `spring.sql.init.mode=always`라서 DB 스키마를 한 곳에서 관리하기 좋다.

## Ingest Flow
- 먼저 FastAPI 서버 재시작 후 `/health`로 `database_url` 기반 설정이 로딩되는지 확인한다.
- `/setup/qdrant`로 Qdrant 컬렉션 존재를 확인한다.
- `woori-vault/복지정책/*.md.md` 파일을 하나씩 `/api/upload/markdown/ingest?chunk_limit=200`에 업로드한다.
- 각 응답의 `job_id`를 `/api/upload/jobs/{job_id}`로 조회해서 `completed`인지 확인한다.
- 마지막으로 `/api/chat`에 테스트 질문을 보내 검색/답변이 되는지 확인한다.

## Assumptions
- `DATABASE_URL` 방식을 사용한다. 이유는 FastAPI/SQLAlchemy에서 가장 단순하고, Neon 같은 특정 서비스명에 묶이지 않기 때문이다.
- Oracle VM PostgreSQL 접속 정보는 Spring과 같은 DB를 사용한다.
- Qdrant는 계속 사용한다. PostgreSQL로 벡터를 옮기지는 않는다.
- 기존 Neon 관련 이름은 코드에서 제거하거나 호환용 fallback 없이 정리한다.
