## 목적

공공데이터포털, 경찰청, Safe182 등 외부 API를 사용할 때 공통으로 지켜야 할 문서화 기준과 관리 규칙을 정리한다.

이 문서는 API를 실제로 호출하는 코드 설명이 아니라, 외부 API를 프로젝트에서 관리할 때 지켜야 하는 공통 원칙을 정의한다.

## 기본 원칙

1. API 문서는 원본 명세와 프로젝트 사용 목적을 구분해서 작성한다.

2. 인증키 원문은 문서에 저장하지 않는다.

3. API 응답 전체를 무조건 저장하지 않는다.

4. DB 저장 대상과 RAG 변환 대상은 분리해서 판단한다.

5. 사용하지 않는 API 기능은 문서에 과하게 상세히 작성하지 않는다.

6. 실제 코드 위치나 서비스 처리 흐름은 API 문서가 아니라 프로젝트 구현메모에 작성한다.

## API 문서에 포함할 내용

1. API 이름

2. 데이터 제공 기관

3. 데이터 출처

4. API 용도

5. Endpoint

6. 요청 방식

7. 요청 파라미터

8. 응답 데이터 형식

9. 주요 응답 필드

10. 우리 서비스에서 사용하는 위치

11. RAG 변환 여부

12. 주의사항

## API 문서에 포함하지 않는 내용

1. 실제 인증키 원문

2. Encoding 인증키 원문

3. Decoding 인증키 원문

4. 서버 환경변수 실제 값

5. 개인정보 원문

6. 대량 원본 응답 전체

7. 임시 테스트 로그

8. 실패한 테스트 요청 URL 전체

## 문서 분리 기준

API 자체의 명세는 각 API 폴더에 작성한다.

예시:

01_복지_API  
02_경찰청_API

서버 코드에서 API를 어떻게 호출하는지는 프로젝트 구현메모에 작성한다.

예시:

03_프로젝트_구현메모

API 응답을 Markdown으로 변환하는 기준은 RAG 변환규칙에 작성한다.

예시:

04_RAG_변환규칙

## 공공데이터 API 공통 주의사항

공공데이터포털 API는 보통 serviceKey를 사용한다.

serviceKey는 서버 환경변수 또는 application.properties에서 관리한다.

API 문서에는 serviceKey라는 변수명만 작성하고 실제 값은 작성하지 않는다.

요청 URL 예시를 작성할 때도 실제 인증키를 포함하지 않는다.

## 요청 URL 작성 규칙

인증키가 필요한 요청 URL은 아래처럼 작성한다.

```text
https://example.api.go.kr/exampleEndpoint?serviceKey={SERVICE_KEY}&pageNo=1&numOfRows=10
