## 사용 목적

사회복지시설 목록과 상세 정보를 조회하여 사용자 주변 복지시설 안내, 시설 기반 복지 정보 제공, 지역 기반 복지 추천에 활용한다.

시설 목록, 시설별 기본정보, 행사 정보, 구인 정보 등을 조회하여 사용자와 복지사가 지역 내 복지시설 정보를 확인할 수 있도록 한다.

7. DB 또는 RAG 문서 저장소에 저장
    
8. 필요 시 RAG 검색용 Markdown 문서로 변환
    
9. raw_hash 비교로 변경 여부 확인
    
10. 변경된 문서만 Qdrant에 재임베딩
    

## DB 저장 기준

시설 코드를 기준으로 중복 저장을 방지한다.

```
중복 기준 = fcltCd
```

시설 코드가 없는 경우에는 시설명, 주소, 시설 종류를 조합하여 중복 여부를 판단한다.

시설명, 주소, 시설 종류, 전화번호, 지역 정보 등 서비스 제공에 필요한 필드 중심으로 저장한다.

행사 정보와 구인 정보는 시설 기본정보와 성격이 다르므로 필요 시 별도 테이블로 분리할 수 있다.

현재 복지시설 전용 테이블이 없다면, 우선 RAG 검색용 문서로 변환하여 `rag_documents` 테이블에 저장하고, 수집 작업 이력은 `rag_ingest_jobs` 테이블에서 관리할 수 있다.

추후 화면에서 시설 목록, 상세 조회, 지역 필터링을 직접 제공해야 한다면 별도 테이블을 추가할 수 있다.

예상 추가 테이블:

```
welfare_facilities
welfare_facility_events
welfare_facility_jobs
```

## RAG 변환 기준

시설 데이터는 양이 많을 수 있으므로 전체를 무조건 임베딩하지 않는다.

사용자 지역, 시설 종류, 주요 서비스가 있는 데이터만 선별한다.

시설명, 주소, 연락처처럼 사용자가 실제로 찾을 가능성이 높은 정보 중심으로 변환한다.

행사 정보와 구인 정보는 최신성이 중요하므로 오래된 데이터가 RAG 답변에 섞이지 않도록 갱신 기준을 별도로 관리한다.

RAG 문서는 원본 XML 구조가 아니라 사용자가 이해할 수 있는 설명형 Markdown으로 변환한다.

## 주의사항

인증키 원문은 문서에 저장하지 않는다.

시설 목록 데이터는 양이 많을 수 있으므로 페이지네이션 처리가 필요하다.

지역 코드와 시설 종류 코드를 함께 관리해야 검색 정확도가 높아진다.

시설별 기본정보 조회에는 시설 코드가 필요할 수 있으므로 목록 조회 결과와 상세 조회를 연계해야 한다.

API 응답 필드는 비어 있을 수 있으므로 null 처리가 필요하다.

RAG에는 시설 원본 데이터 전체보다 사용자가 질문할 만한 요약 정보를 저장한다.

행사 정보와 구인 정보는 시간이 지나면 오래된 정보가 될 수 있으므로 동기화 주기와 만료 기준을 따로 관리한다.

## 최종 정리

이 API는 사용자에게 주변 사회복지시설을 안내하고 지역 기반 복지 정보를 제공하기 위한 데이터 소스다.

DB에는 시설 코드 기준으로 주요 정보를 저장하고, RAG에는 사용자가 검색할 가능성이 높은 시설명, 주소, 시설 종류, 연락처, 주요 서비스 중심으로 변환한다.

시설 행사 정보와 구인 정보는 최신성이 중요하므로 필요 시 별도 테이블 또는 별도 RAG 문서로 관리한다.
